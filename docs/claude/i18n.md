# Claude: carregar para tarefas de internacionalização (i18n)

## Regras obrigatórias

1. **Nunca texto hardcoded no JSX** — qualquer string visível usa `t()` ou `tc()`.
2. Após implementar novas chaves: editar o ficheiro `translations/<namespace>.json` nos 4 idiomas e correr `npm run seed` no container backend.

## Declaração de namespace

```typescript
const { t } = useTranslation('nome_do_namespace');  // específico da página
const { t: tc } = useTranslation('common');          // chaves partilhadas
```

## Namespaces registados

Listados em `frontend/src/i18n/index.ts` e semeados via `backend/prisma/seeds/translations/`:
`common`, `auth`, `dashboard`, `users`, `teams`, `projects`, `plans`, `holidays`,
`planning`, `permissions`, `translations`, `board`, `calendar`, `gantt`, `sessions`,
`notifications`, `timesheet`.

> Cada ferramenta self-contained tem o seu namespace (ex.: `board`, `gantt`),
> mesmo quando a UI vive dentro doutra página (ex.: `gantt` é tab dentro do
> Planning). Isto torna a gestão de traduções no backoffice independente e
> evita que o namespace `planning` cresça sem limites.

## Estrutura dos ficheiros de seed

Cada namespace tem o seu próprio ficheiro em `backend/prisma/seeds/translations/`:

```
prisma/seeds/translations/
  auth.json
  board.json
  calendar.json
  common.json
  dashboard.json
  gantt.json
  holidays.json
  notifications.json
  permissions.json
  planning.json
  plans.json
  projects.json
  sessions.json
  teams.json
  timesheet.json
  translations.json
  users.json
```

Formato de cada ficheiro (`locale → { key → value }`):

```json
{
  "en":    { "page.title": "Title",  "btn.save": "Save"    },
  "es":    { "page.title": "Título", "btn.save": "Guardar" },
  "pt-BR": { "page.title": "Título", "btn.save": "Salvar"  },
  "pt-PT": { "page.title": "Título", "btn.save": "Guardar" }
}
```

## Adicionar chaves a um namespace existente

Editar directamente o ficheiro correspondente em `prisma/seeds/translations/`:

```json
// prisma/seeds/translations/my_namespace.json
{
  "en":    { ..., "new.key": "New value"  },
  "es":    { ..., "new.key": "Nuevo valor" },
  "pt-BR": { ..., "new.key": "Novo valor" },
  "pt-PT": { ..., "new.key": "Novo valor" }
}
```

Depois executar no container backend:
```bash
docker exec awesome-project-app-backend npm run seed
```

## Criar novo namespace (nova página/funcionalidade)

1. Criar `backend/prisma/seeds/translations/novonamespace.json` com os 4 locales
2. Adicionar `'novonamespace'` ao array `ns` em `frontend/src/i18n/index.ts`
3. Correr seed no container: `docker exec awesome-project-app-backend npm run seed`
4. Usar `useTranslation('novonamespace')` no componente React

## Workflow de Export Seed (backoffice → git)

Após editar traduções no backoffice `/translations` e querer persistir em ficheiro:

1. Backoffice → "⬇ Exportar Seed" → guarda `translations.json` localmente
2. Copiar para `backend/prisma/seeds/translations.json`
3. Correr split no container: `docker exec awesome-project-app-backend node scripts/split-translations.js`
4. Commit apenas dos ficheiros em `prisma/seeds/translations/` que mudaram
5. Apagar `prisma/seeds/translations.json` (artefacto temporário — não commitar)

> O script `split-translations.js` processa apenas o Format A (namespace → locale → key)
> e ignora blocos locale-keyed (Format B). O Format A é o que o endpoint `/api/i18n/export`
> gera. O seed v2.0 lê directamente do directório `translations/`.

## Cross-namespace lookup

Para usar uma chave doutro namespace sem criar nova `useTranslation()`,
prefixar com `'<namespace>:'`:

```typescript
const { t } = useTranslation('planning'); // ns principal da página
t('common:errors.forbidden');             // chave do namespace common
t('gantt:gantt.today_marker');            // chave do namespace gantt
```

Útil em handlers dentro de `tRef.current(...)` (DHTMLX events) onde adicionar
um segundo `tRef` por namespace seria excessivo.

## Convenção de nomes de chaves

| Prefixo | Uso |
|---------|-----|
| `page.title` / `page.subtitle` | Cabeçalho da página |
| `btn.add` / `btn.new_x` | Botões de acção primária |
| `modal.create_title` / `modal.edit_title` | Títulos de modais |
| `modal.delete.title` / `modal.delete.warning` | Confirmação de eliminação |
| `form.field` / `form.field_placeholder` | Campos de formulário |
| `table.col_name` | Colunas específicas da página |
| `stats.total` / `stats.active` | Contadores/badges |
| `success.created` / `success.updated` / `success.deleted` | Feedback positivo |
| `search.placeholder` / `search.empty` | Pesquisa |
| `empty.no_items` / `empty.no_match` | Estados vazios |
| `feature.unavailable_title` / `feature.unavailable_text` | Plano insuficiente |
| `x.col.*` / `x.btn.*` / `x.form.*` | Sub-grupos dentro dum contexto |

## Anti-padrões

- ❌ Texto hardcoded em JSX (labels, placeholders, tooltips, erros, confirmações)
- ❌ Criar chaves em `common` para strings usadas numa só página
- ❌ Commitar `prisma/seeds/translations.json` — é artefacto de export, não fonte de verdade
- ❌ Editar `translations.json` directamente — editar sempre o ficheiro de namespace em `translations/`
- ❌ Conflito leaf-vs-parent no mesmo namespace:

```
✗  members.status = "Status"  +  members.status.invited = "Invited"   ← 500 na API
✓  members.col.status = "Status"  +  members.status.invited = "Invited"
```
