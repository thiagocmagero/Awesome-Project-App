## Visão Geral

A aplicação usa **react-i18next** para internacionalização completa. Toda a UI — labels, mensagens de erro, placeholders, tooltips, títulos de página, textos de confirmação, toasts, SweetAlerts — deve usar o sistema de traduções.

As traduções são geridas num **backoffice dedicado** dentro do painel (`/translations`), acessível apenas a `PLATFORM_ADMIN`. O admin edita, revê e aprova traduções directamente na UI, com suporte a **tradução automática via Claude API** (client-side, sem chaves no servidor).

**Princípio fundamental:** chaves sem tradução **nunca fazem fallback silencioso** para outro locale. A chave crua aparece na UI (ex: `users.page.title`) como alerta visual de que algo está em falta. O backoffice rastreia e sinaliza todas as chaves em falta.

**Rotas do frontend não mudam.** O idioma é uma preferência do utilizador guardada em `localStorage` — não faz parte da URL. As rotas permanecem `/users`, `/projects`, `/translations`, etc.

## Stack de Internacionalização

### Dependências Frontend
```bash
npm install react-i18next i18next i18next-http-backend i18next-browser-languagedetector
```

| Pacote | Papel |
|--------|-------|
| `i18next` | Motor de internacionalização |
| `react-i18next` | Integração React (hooks, HOC, Suspense) |
| `i18next-http-backend` | Carrega traduções via HTTP da API em runtime |
| `i18next-browser-languagedetector` | Detecta locale do browser automaticamente |

### Sem ficheiros JSON locais de tradução
As traduções **não vivem em ficheiros `.json` locais**. São servidas pela API backend em `GET /api/i18n/:locale/:namespace` e guardadas na base de dados. O `i18next-http-backend` faz o fetch automático ao inicializar. A fonte de verdade versionada é o ficheiro `backend/prisma/seeds/translations.json` (ver secção de persistência).

## Locales Suportados e Fallback Chain

Os locales são geridos dinamicamente pela tabela `Locale` na base de dados. O `PLATFORM_ADMIN` pode adicionar novos locales pelo backoffice (`/translations` → tab "Idiomas"). **Novos locales iniciam como INACTIVE** — não aparecem no LanguageSelector da aplicação até serem activados manualmente pelo admin.

### Locales built-in (seed inicial — `isBuiltIn: true`)

| Código | Nome | Flag | Estado inicial |
|--------|------|------|---------------|
| `pt-PT` | Português (Portugal) | `portugal_flag.jpg` | ACTIVE |
| `pt-BR` | Português (Brasil) | `brazil_flag.jpg` | ACTIVE |
| `en` | English | `us_flag.jpg` | ACTIVE |
| `es` | Español | `spain_flag.jpg` | ACTIVE |

Locales built-in **não podem ser eliminados**. Podem ser desactivados (ex: ocultar pt-BR temporariamente).

### Configuração de fallback no i18next
```typescript
fallbackLng: 'en',  // qualquer locale desconhecido → English
```

O fallback é simples por design — locales desconhecidos (adicionados dinamicamente) recaem sempre em `en`. O objectivo é ter **100% das chaves traduzidas em todos os locales activos** antes de os activar. O backoffice rastreia e sinaliza todas as chaves em falta.

### Detecção automática de locale
O `i18next-browser-languagedetector` detecta automaticamente o locale do browser. O utilizador pode substituir manualmente via selector de idioma no header (apenas locales ACTIVE são mostrados). A preferência é guardada em `localStorage` com a chave `i18n_locale`.

### `supportedLngs` — não definido estaticamente
O `supportedLngs` **não é definido** na configuração i18next — os locales são dinâmicos. O `LanguageSelector` consulta `GET /api/i18n/locales/active` para saber que locales mostrar.

## Configuração Base

### `frontend/src/i18n/index.ts`
```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import HttpBackend from 'i18next-http-backend';
import LanguageDetector from 'i18next-browser-languagedetector';

i18n
  .use(HttpBackend)
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    fallbackLng: 'en',  // qualquer locale desconhecido → English (locales são dinâmicos)
    // supportedLngs: NÃO definido — locales são geridos dinamicamente pela tabela Locale
    defaultNS: 'common',
    ns: [
      'common',
      'auth',
      'dashboard',
      'users',
      'teams',
      'projects',
      'plans',
      'holidays',
      'planning',
      'translations',
    ],
    backend: {
      loadPath: '/api/i18n/{{lng}}/{{ns}}',
    },
    interpolation: {
      escapeValue: false,
    },
    detection: {
      order: ['localStorage', 'navigator'],
      caches: ['localStorage'],
      lookupLocalStorage: 'i18n_locale',
    },
    // Chaves sem tradução mostram a chave crua — nunca string vazia
    parseMissingKeyHandler: (key: string) => key,
    // Reporta chaves em falta ao backend para rastreio no backoffice
    saveMissing: true,
    saveMissingTo: 'all',
    missingKeyHandler: (lngs: string[], ns: string, key: string) => {
      const token = localStorage.getItem('token');
      if (!token) return;
      fetch('/api/i18n/missing', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          Authorization: `Bearer ${token}`,
        },
        body: JSON.stringify({ locale: lngs[0], namespace: ns, key }),
      }).catch(() => {}); // fire-and-forget, nunca bloquear a UI
    },
  });

export default i18n;
```

## Convenções de Chaves de Tradução

### Estrutura hierárquica
```
namespace:section.subsection.key
```

O namespace é declarado no `useTranslation('namespace')` — não é prefixado nas chaves dentro do `t()`.

### Namespaces por módulo

| Namespace | Conteúdo |
|-----------|----------|
| `common` | Botões globais, status, paginação, confirmações, navegação sidebar, erros genéricos |
| `auth` | Login, logout, registo, erros de autenticação |
| `dashboard` | KPIs, textos do dashboard |
| `users` | UsersPage, UserTypesPage, UserLevelsPage — modais, tabelas, campos, erros |
| `teams` | TeamsPage — modais, tabelas, campos, erros |
| `projects` | ProjectsPage — modais, tabelas, campos, erros, membros, feriados associados |
| `plans` | PlansPage — tabs, limites, pricing, feature flags |
| `holidays` | HolidaysPage — modais, tabelas, campos, datas |
| `planning` | Planning/Gantt — tarefas, recursos, dependências, mensagens Gantt |
| `translations` | Backoffice de traduções — meta-UI (o próprio backoffice traduzido) |

### Regras de nomenclatura de chaves

1. **snake_case** para todas as chaves — nunca camelCase nem kebab-case.
2. **Máximo 3 níveis** de hierarquia: `section.subsection.key` (ex: `modal.fields.name`).
3. **Sem abreviações** — `btn_cancel` não, `actions.cancel` sim.
4. **Pluralização** via sufixo i18next nativo:
   ```typescript
   // Chaves: "count_one": "{{count}} utilizador", "count_other": "{{count}} utilizadores"
   t('users:count', { count: n })
   ```
5. **Interpolação** via `{{variavel}}`:
   ```typescript
   // Chave: "welcome": "Bem-vindo, {{name}}!"
   t('auth:welcome', { name: user.name })
   ```
6. **Nunca hardcode de texto** nos componentes — toda string visível ao utilizador vai para tradução, incluindo `placeholder`, `title`, `aria-label`, `alt`.
7. **Erros do backend** — o backend devolve `error_code` (SCREAMING_SNAKE_CASE) em vez de mensagem em linguagem natural. O frontend traduz via `t('common:errors.email_already_exists')`. Ver secção de convenção de error codes.

### Convenção de error codes do backend

O backend **nunca** devolve mensagens de erro em linguagem natural. Devolve sempre:
```json
{ "error_code": "EMAIL_ALREADY_EXISTS", "statusCode": 409 }
```
O frontend mapeia:
```typescript
const msg = t(`common:errors.${error.error_code.toLowerCase()}`);
showToast('danger', msg);
```
Chaves de erro obrigatórias no namespace `common`:
```
errors.email_already_exists
errors.invalid_credentials
errors.forbidden
errors.not_found
errors.plan_limit_reached
errors.feature_not_available
errors.validation_failed
errors.generic
```

## Regras Obrigatórias — Novo Código

### Regra 1 — `useTranslation` em todos os componentes com texto
```typescript
import { useTranslation } from 'react-i18next';

const UsersPage = () => {
  const { t } = useTranslation('users');
  const { t: tc } = useTranslation('common'); // alias quando se usa common também

  return (
    <>
      <h1>{t('page.title')}</h1>
      <button>{tc('actions.save')}</button>
    </>
  );
};
```

Alternativa com múltiplos namespaces num único hook:
```typescript
const { t } = useTranslation(['users', 'common']);
// Namespace explícito no t():
t('users:page.title')
t('common:actions.save')
```

## Regras Obrigatórias — Código Existente (Retrofit)

Quando instruído para **chavear** (retrofit) uma página ou componente existente, seguir **obrigatoriamente** este processo:

### Processo de Retrofit — 5 Passos

**Passo 1 — Inventário**
Listar todos os textos hardcoded: labels de tabela, títulos de modal, placeholders, botões, mensagens de erro inline, toasts, SweetAlerts, breadcrumbs, `document.title`.

**Passo 2 — Mapeamento de chaves**
Para cada texto, definir a chave seguindo `namespace:section.key`:
```
"Utilizadores"                → users:page.title
"Gestão de utilizadores"      → users:page.subtitle
"Novo Utilizador"             → users:modal.create_title
"Nome completo"               → users:modal.fields.name_placeholder
"Guardar"                     → common:actions.save
"Registo criado com sucesso." → common:messages.success_created
```

**Passo 3 — Seed no backend**
Adicionar todas as chaves novas ao `seedKeys()` do `I18nService` para todos os locales built-in (pt-PT, pt-BR, en, es). Novos locales adicionados dinamicamente recebem os registos MISSING automaticamente via `createLocale()`.

**Passo 4 — Substituição no componente**
- Adicionar `const { t } = useTranslation('namespace')` no topo.
- Substituir cada string literal pelo `t('chave')` correspondente.
- Verificar `placeholder`, `title`, `aria-label`, `Swal.fire`, `showToast`, `document.title`.
- Substituir `throw new Error(data.message)` por `throw data` + `handleApiError(err, t, showToast)`.

**Passo 5 — Verificação**
Confirmar que não sobrou nenhuma string visível ao utilizador hardcoded. Mudar temporariamente o locale para `en` e verificar que tudo muda — qualquer texto que não mude está hardcoded.

### Estado do Retrofit por Página

## Backend — Persistência e Migração de Traduções

### Estratégia: Tabela Translation + Seed Versionado em Git

As traduções vivem na base de dados de cada ambiente (dev, QA, prod). A fonte de verdade versionada é o ficheiro `backend/prisma/seeds/translations.json`, exportado pelo backoffice e commitado em git. O seed é idempotente (upsert com `update: {}`) — nunca sobrescreve traduções já editadas pelo admin.

**Fluxo de propagação entre ambientes:**
```
Admin edita/aprova traduções em PROD
        ↓
Backoffice: botão "⬇ Exportar Seed"
        ↓
GET /api/i18n/export → devolve translations.json com todas as chaves APPROVED
        ↓
Admin faz download e substitui backend/prisma/seeds/translations.json
        ↓
git commit + push
        ↓
Em QA / dev: git pull → npm run seed
        ↓
upsert idempotente insere chaves novas, nunca toca nas já existentes
```

### Modelos Prisma

```prisma
/// Locale suportado pela aplicação — gerido pelo PLATFORM_ADMIN
model Locale {
  id        Int      @id @default(autoincrement())
  code      String   @unique   // "pt-PT" | "pt-BR" | "en" | "es" | "fr" | ...
  name      String             // "Português (Portugal)" | "English" | ...
  flag      String?            // "portugal_flag.jpg" (em /assets/images/flags/)
  isActive  Boolean  @default(false)  // false = não aparece no LanguageSelector
  isBuiltIn Boolean  @default(false)  // true = não pode ser eliminado
  order     Int      @default(0)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

enum TranslationStatus {
  APPROVED      // tradução confirmada pelo admin (manual ou aprovação de sugestão IA)
  AI_SUGGESTED  // tradução gerada pela IA, aguarda revisão do admin
}

/// Tradução de uma chave num namespace para um locale
model Translation {
  id        Int               @id @default(autoincrement())
  publicId  String            @unique @default(uuid(7))
  locale    String            // "pt-PT" | "pt-BR" | "en" | "es" | ...
  namespace String            // "common" | "users" | "teams" | ...
  key       String            // "actions.save" | "page.title" | ...
  value     String            // texto traduzido (vazio string = MISSING)
  status    TranslationStatus @default(APPROVED)
  createdAt DateTime          @default(now())
  updatedAt DateTime          @updatedAt

  @@unique([locale, namespace, key])
  @@index([locale, namespace])
  @@index([status])
}

/// Chaves reportadas como em falta pelo frontend (missingKeyHandler do i18next)
model MissingTranslation {
  id        Int      @id @default(autoincrement())
  locale    String
  namespace String
  key       String
  seenAt    DateTime @default(now())
  resolved  Boolean  @default(false)

  @@unique([locale, namespace, key])
  @@index([resolved])
}
```

### Seed dos Locales Built-in

```javascript
// seed.js — upsert idempotente, update:{} preserva alterações do admin
const BUILTIN_LOCALES = [
  { code: 'pt-PT', name: 'Português (Portugal)', flag: 'portugal_flag.jpg', isActive: true, isBuiltIn: true, order: 0 },
  { code: 'pt-BR', name: 'Português (Brasil)',   flag: 'brazil_flag.jpg',   isActive: true, isBuiltIn: true, order: 1 },
  { code: 'en',    name: 'English',              flag: 'us_flag.jpg',       isActive: true, isBuiltIn: true, order: 2 },
  { code: 'es',    name: 'Español',              flag: 'spain_flag.jpg',    isActive: true, isBuiltIn: true, order: 3 },
];
for (const loc of BUILTIN_LOCALES) {
  await prisma.locale.upsert({
    where: { code: loc.code },
    create: loc,
    update: {}, // preserva alterações do admin (ex: desactivar pt-BR)
  });
}
```

### Endpoints do `I18nController`

**Ordem obrigatória das rotas** (NestJS resolve top-down — rotas específicas ANTES de rotas com parâmetros):

```typescript
// ─── LOCALES (PÚBLICO + PLATFORM_ADMIN) ──────────────────────────────────────

// GET /api/i18n/locales/active
// Retorna apenas locales isActive=true — usado pelo LanguageSelector (sem auth)
@Get('locales/active')
getActiveLocales(): Promise<Locale[]>

// GET /api/i18n/locales
// Retorna todos os locales com stats (total_keys, missing, ai_suggested) — PLATFORM_ADMIN
@Get('locales')
@UseGuards(JwtAuthGuard, ProfilesGuard)
@RequireProfiles('PLATFORM_ADMIN')
getAllLocales(): Promise<LocaleWithStats[]>

// POST /api/i18n/locales
// Cria novo locale (isActive=false) + auto-gera Translation MISSING para todas as chaves existentes
@Post('locales')
@UseGuards(JwtAuthGuard, ProfilesGuard)
@RequireProfiles('PLATFORM_ADMIN')
createLocale(dto: CreateLocaleDto): Promise<Locale>

// PATCH /api/i18n/locales/:code
// Actualiza name, flag, isActive, order — isBuiltIn não pode ser alterado
@Patch('locales/:code')
@UseGuards(JwtAuthGuard, ProfilesGuard)
@RequireProfiles('PLATFORM_ADMIN')
updateLocale(code: string, dto: UpdateLocaleDto): Promise<Locale>

// DELETE /api/i18n/locales/:code
// Elimina locale + todas as Translation desse locale — só se !isBuiltIn
@Delete('locales/:code')
@UseGuards(JwtAuthGuard, ProfilesGuard)
@RequireProfiles('PLATFORM_ADMIN')
deleteLocale(code: string): Promise<void>


// ─── PÚBLICOS (usados pelo i18next-http-backend) ─────────────────────────────

// GET /api/i18n/:locale/:namespace
// Retorna { "actions.save": "Guardar", ... } — apenas chaves APPROVED com value não vazio
@Get(':locale/:namespace')
getNamespace(locale: string, namespace: string): Promise<Record<string, string>>

// POST /api/i18n/missing  (JWT — evitar spam anónimo)
// Regista chave em falta para rastreio no backoffice
@Post('missing')
@UseGuards(JwtAuthGuard)
reportMissing(dto: ReportMissingDto)
```

### Comportamento de Edição por Estado de Célula

**Célula APPROVED (🟢):**
- Mostra o valor traduzido.
- Em hover: aparece botão ✨ (traduzir com IA).
- Click → textarea de edição inline.
- `Enter` / blur → guarda, status permanece APPROVED.
- `Escape` → cancela sem guardar.

**Célula AI_SUGGESTED (🟡):**
- Fundo amarelo âmbar suave.
- Mostra o valor sugerido pela IA.
- Botões `✓ Aprovar` e `✗ Rejeitar` sempre visíveis (não apenas em hover).
- Click na célula → edição inline; guardar muda status para APPROVED.
- `✓ Aprovar` → status APPROVED, fundo normaliza.
- `✗ Rejeitar` → apaga value, status MISSING (fundo vermelho).

**Célula MISSING (🔴):**
- Fundo vermelho suave, sem valor.
- Botão ✨ sempre visível.
- Click → textarea de edição inline; guardar muda status para APPROVED.

## Tradução por IA — Claude API

### Princípio de Funcionamento

O backoffice chama a API Anthropic **directamente do browser** (client-side). Não é necessário guardar chaves no servidor nem criar endpoints proxy. O modelo usado é sempre `claude-sonnet-4-20250514`.

```typescript
const callClaudeAPI = async (prompt: string): Promise<string> => {
  const response = await fetch('https://api.anthropic.com/v1/messages', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      model: 'claude-sonnet-4-20250514',
      max_tokens: 1000,
      messages: [{ role: 'user', content: prompt }],
    }),
  });
  const data = await response.json();
  return data.content[0].text.trim();
};
```

### Tradução Individual (botão ✨ por célula)

```typescript
const translateSingleCell = async (
  locale: string,
  namespace: string,
  key: string,
  allTranslationsForKey: Record<string, string>,
) => {
  setCellLoading(locale, key, true);

  try {
    const prompt = buildTranslationPrompt({
      targetLocale: locale,
      namespace,
      key,
      existingTranslations: allTranslationsForKey,
    });

    const suggestion = await callClaudeAPI(prompt);

    // Guarda no backend com status AI_SUGGESTED
    await fetch(`/api/i18n/${locale}/${namespace}/${encodeURIComponent(key)}`, {
      method: 'PATCH',
      headers: {
        'Content-Type': 'application/json',
        Authorization: `Bearer ${token}`,
      },
      body: JSON.stringify({ value: suggestion, status: 'AI_SUGGESTED' }),
    });

    // Actualiza state local → célula passa a 🟡
    updateCell(locale, key, suggestion, 'AI_SUGGESTED');
    refreshStats();
  } catch {
    showToast('danger', t('translations:errors.ai_failed'));
  } finally {
    setCellLoading(locale, key, false);
  }
};
```

### Tradução em Massa — "✨ Traduzir tudo em falta"

Traduz todas as células MISSING do namespace activo sequencialmente com delay entre chamadas.

```typescript
const translateAllMissing = async () => {
  const missingCells = getMissingCells(); // [{ locale, key, existingTranslations }]
  const cancelRef = { current: false };

  setIsBulkTranslating(true);
  setBulkCancelRef(cancelRef);
  showToast('info', t('translations:messages.bulk_started', { count: missingCells.length }));

  for (let i = 0; i < missingCells.length; i++) {
    if (cancelRef.current) break;

    const cell = missingCells[i];
    await translateSingleCell(cell.locale, activeNamespace, cell.key, cell.existingTranslations);
    setBulkProgress({ current: i + 1, total: missingCells.length });

    // 300ms entre chamadas — respeita rate limits da API
    if (i < missingCells.length - 1 && !cancelRef.current) {
      await new Promise(resolve => setTimeout(resolve, 300));
    }
  }

  setIsBulkTranslating(false);

  if (!cancelRef.current) {
    showToast('success', t('translations:messages.bulk_complete', { count: missingCells.length }));
    // Redireciona automaticamente para sub-tab "Por Rever"
    setActiveSubTab('ai_suggested');
  }
};
```

```typescript
// Aprovar → APPROVED, invalida MissingTranslation, recarrega namespace no i18next
const approveCell = async (locale: string, key: string) => {
  await fetch(`/api/i18n/backoffice/${activeNamespace}/approve`, {
    method: 'PATCH',
    headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },

// Rejeitar → MISSING, apaga valor

    method: 'PATCH',
    headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
    body: JSON.stringify({ value: '', status: 'MISSING' }),
  });
  updateCell(locale, key, '', 'MISSING');
  refreshStats();
};
```

## Notas Finais para o Claude Code

1. **`import './i18n'` deve ser sempre o primeiro import em `main.tsx`** — antes de React, antes de qualquer componente.

2. **`<Suspense>` é obrigatório** — as traduções são assíncronas (fetch da API). Sem Suspense a UI renderiza com chaves cruas antes do fetch completar.

3. **`parseMissingKeyHandler: (key) => key`** — garante que chaves em falta mostram a chave crua (ex: `users.page.title`) na UI. É o comportamento pretendido: alerta visual imediato.

4. **`missingKeyHandler` é fire-and-forget** — nunca `await`, nunca bloquear a UI. O `.catch(() => {})` é intencional.

5. **`update: {}` no upsert do seed** — nunca sobrescreve traduções existentes. O seed só insere chaves que ainda não existem na BD.

6. **Chaves são imutáveis no código** — renomear uma chave requer: criar a nova no backend, migrar valores existentes via script, actualizar todos os `t('chave')` no código, eliminar a antiga. Nunca renomear directamente.

7. **Error codes do backend** — o backend nunca devolve mensagens de erro em linguagem natural. Sempre `error_code` em SCREAMING_SNAKE_CASE. O frontend traduz via `t(\`common:errors.${code.toLowerCase()}\`)`.

8. **Após aprovar uma tradução** — chamar `i18n.reloadResources(locale, namespace)` para reflectir a mudança na UI sem reload de página.

9. **Exportar seed antes de cada promoção para QA/Prod** — o ficheiro `translations.json` em git é a fonte de verdade para propagação entre ambientes.

10. **Namespace `translations` é auto-referencial** — o backoffice de traduções usa `useTranslation('translations')`. Deve ser o último namespace a ser retrofitado.

11. **`AppLayout.tsx` deve ser o primeiro retrofit** — contém a navegação sidebar e header partilhados por toda a aplicação. Sem ele, o `common:nav.*` aparece como chave crua em todos os ecrãs.

12. **Modelo `Translation` segue convenções do projecto** — tem `publicId` (UUID v7), `createdAt`, `updatedAt`. O `id` numérico é interno e nunca exposto. Controllers usam `publicId` quando necessário.
